# MatrixBHN — Private Communications Network

MatrixBHN is a self-hosted [Matrix](https://matrix.org) homeserver running on the Blackhole Network LA hub, paired with a WireGuard VPN mesh that extends to close friends and family. The goal is a small, invitation-only network where members get two things simultaneously: a private, federated-free chat platform and a personal VPN with DNS-level ad/tracker blocking.

---

## Why This Exists

Most messaging platforms — even end-to-end encrypted ones — are run by third parties. Your metadata (who you talk to, when, how often) is logged by the provider even when message content is protected. Signal is excellent but requires a phone number. WhatsApp is Meta. iMessage requires Apple accounts. Discord is a surveillance product dressed as a chat app.

Matrix is an open protocol with a clean threat model: when you self-host the homeserver, you control the server logs, the user database, the retention policy, and the encryption keys. There is no company that can be compelled to hand over your data because there is no company — just a server you own.

The additional benefit here is the WireGuard overlay. Every member connects through a VPN tunnel before reaching the Matrix server, which means:
- DNS queries are resolved by AdGuard (ad/tracker blocking for all network traffic, not just the browser)
- The Matrix server is completely unreachable from the public internet — no attack surface beyond the WireGuard port
- Members get a private subnet they can use for anything that benefits from a trusted network path

## Architecture

```
[Member device]
      │
      │  WireGuard tunnel (encrypted UDP)
      ▼
[BHN LA Hub]
  ├── AdGuard Home  — DNS resolver (blocks ads, trackers, malware domains)
  ├── Matrix Synapse  — homeserver, WireGuard interface only
  │     HTTP  port 8008  — desktop Element client
  │     HTTPS port 8448  — mobile Element client (self-signed cert)
  └── WireGuard  — mesh hub, all member peers terminate here
```

The Matrix server binds exclusively to the WireGuard interface. It does not listen on the public internet — there is no port forwarding, no reverse proxy exposed externally, no federation with the broader Matrix network. If you're not on the VPN, the server doesn't exist.

Federation is intentionally disabled. MatrixBHN is a closed network. You cannot receive messages from or send messages to `@anyone:matrix.org` or any other homeserver. This is by design — federation would extend trust to servers outside our control.

## WireGuard Mesh — How Guest Access Works

New members receive a WireGuard configuration file generated by the operator. By default it is a **split-tunnel** configuration:

- Only traffic destined for the BHN private subnet routes through the tunnel
- DNS queries route through the tunnel (AdGuard protection applies to all DNS lookups regardless of where traffic goes)
- All other traffic (streaming, browsing, work VPN) uses the member's own ISP directly — no performance impact on normal internet use

Full-tunnel mode (all traffic through BHN) is available on request, which routes everything through the LA hub's upstream and applies AdGuard to all DNS globally.

The mesh uses standard WireGuard with pre-shared keys per peer for an additional layer of post-quantum resistance. Each guest peer gets a static IP in the private subnet.

## Matrix Client Setup

**Recommended client:** [Element](https://element.io/download) — available on iOS, Android, macOS, Windows, Linux, and browser.

Connection requires an active WireGuard connection first. Once on the VPN:

- **Desktop Element** — point to the HTTP endpoint (no certificate prompt)
- **Mobile Element** — point to the HTTPS endpoint; accept the self-signed certificate once on first connect (the cert is valid for 10 years and is pinned to the VPN IP)

The DNS alias `chat.bhn.local` resolves automatically once connected (via AdGuard). Accounts are created by the operator — there is no self-registration.

## What's Running

| Component | Details |
|-----------|---------|
| Homeserver | Matrix Synapse (Python), native install |
| Database | SQLite on LUKS2-encrypted NVMe |
| Media store | LUKS2-encrypted NVMe |
| Certificates | Self-signed 4096-bit RSA, SAN bound to VPN IP |
| Federation | Disabled |
| Registration | Disabled (operator creates accounts) |
| DNS | AdGuard Home — custom blocklists + BHN local DNS rewrites |

All persistent data (database, media uploads, encryption keys) lives on an AES-256-XTS encrypted NVMe volume. The volume is unlocked at boot with a LUKS2 passphrase stored in Proton Pass; if the server is seized without the passphrase, the data is unreadable.

## Getting Access

MatrixBHN is currently invite-only — access is extended personally to friends and family of the operator. If you were pointed here, reach out directly to get a WireGuard config and account credentials.

There are no plans to open registration publicly. The value of this network is precisely that it's small, trusted, and fully within our control.

## Why Not Just Use Signal / Telegram / etc.?

| Platform | Server control | Metadata | Federation | Self-hostable |
|----------|---------------|----------|------------|---------------|
| Signal | Signal Foundation | Minimal, but exists | No | No |
| Telegram | Telegram (Dubai) | Extensive on cloud chats | No | No |
| WhatsApp | Meta | Extensive | No | No |
| Discord | Discord Inc. | Extensive | No | No |
| **MatrixBHN** | **You** | **None beyond what you log** | **Disabled by choice** | **Yes — this is it** |

The comparison isn't that MatrixBHN is more convenient. It isn't. The comparison is that it's the only option where the infrastructure is physically ours, the logs are ours to set or not set, and there is no third party in the chain.
