WHAT IT DOES (5 sections, all in one BEGIN/COMMIT):

   8 UPDATE statements bringing live tier_labels into sync with the

              + reholder_fee_min_usd NUMERIC(8,2)

   Black Label 10 (rarest) > Gold Label 10 (middle) > Pristine 10 (lowest)

     BGS | Black Label 10 | tier='Black Label' | numeric=10
     BGS | Gold Label 10  | tier='Gold Label'  | numeric=10
   DELETE of 'Gold 10' placeholder from db93ccf (wrong naming convention).

D. RAW grader sentinel + UNKNOWN grader sentinel
   INSERT: RAW | Ungraded   | NULL | Ungraded   | FALSE | FALSE
   INSERT: UNKNOWN | Unparseable | NULL | Unparseable | FALSE | FALSE

E. Fact-table CHECK constraints accept RAW + UNKNOWN
   ebay_transactions, courtyard_transactions, courtyard_asks,
   collector_crypt_transactions, collector_crypt_asks — all 5 get
   the same allowed list: {CGC, PSA, BGS, SGC, RAW, UNKNOWN} + NULL.

CGC LABEL COLOR SCHEME (authoritative — replaces prior wrong drafts):

  Gold   = Pristine 10 (current top tier, virtually flawless, 50/50 centering)
  Black  = current standard CGC (Gem Mint 10, Mint+ 9.5, grades 9 and below)
  Blue   = LEGACY — older naming convention, Gem Mint 9.5 only (outlier)
  (legacy retired) = Perfect 10 (retired 2023)

There is NO current CGC Green Label for trading cards. Earlier session
drafts referenced "Green Label" — all wrong, all removed.

CRITICAL PRICING DISTINCTION (HORIZON load-bearing):

At numeric grade 9.5, two raw_labels exist with very different market
behavior:

  Mint+ 9.5     (Black Label, current standard) — NOT equivalent to a 10.
                Completely different card from Gem Mint 9.5.
  Gem Mint 9.5  (Blue Label, legacy outlier)    — CGC officially treats as
                EQUIVALENT to Gem Mint 10. Eligible for $5–$10 reholder to
                Gem Mint 10.

HORIZON must JOIN on (grader, raw_label) — not (grader, numeric_grade) — to
distinguish these. tier_label is the discriminator at the same numeric.

LATE-SESSION ADDENDUM — TLS FINGERPRINT BREAKTHROUGH (impers + firefox144)
================================================================================

Confirmed empirically tonight from LA's burned IP 149.28.91.100:

1. All three of our data-center IPs return HTTP 403 in <1 second on eBay's
   /sch/i.html search endpoint:
     LA direct  (Vultr LA, 149.28.91.100)         → 403
     FRA tunnel (Vultr Frankfurt, 192.248.187.208) → 403
     Hillsboro  (Hetzner US-WEST, 5.78.94.237)    → 403
   eBay broad-blocks data-center ranges on this endpoint.

2. BUT — the dominant block signal is not IP reputation alone; it's also
   TLS fingerprint. eBay's anti-bot (Akamai-style) scores on JA3 / HTTP-2
   fingerprint AND IP. Standard clients (curl, Node fetch/undici, Python
   requests) emit non-browser TLS handshakes that flag fast.

3. Tested 8 browser-impersonation profiles via the npm package `impers`
   (Node-native TypeScript port of curl_cffi) from LA's burned IP:
     chrome (default, older) → 403 (1.8 KB Error Page)
     chrome142/131/116       → 200 / 13 KB  (Akamai challenge, NOT listings)
     firefox / firefox144    → 200 / 1.47-1.50 MB  (REAL listings page)
     safari / edge           → 200 / 13 KB  (challenge page, NOT listings)

   Firefox is the only profile that gets through to the full search-results
   payload. ALL from LA's burned IP — NO proxy involved.

4. Sustainability test (60s cool + 5 requests × 20s gaps, firefox144):
     [1] 403   1,989 bytes  (residual rate-limit from prior 8-profile batch)
     [2] 200   1.50 MB
     [3] 200   1.50 MB
     [4] 200   1.50 MB
     [5] 200   1.50 MB
   4/5 success. The single 403 was carryover from earlier rapid-fire tests;
   on a fully-cooled IP with paced requests, 5/5 expected.

CONCLUSION

  impers + firefox144 + paced requests = working eBay scrape from LA's
  existing IP. No residential proxies. No managed service. No monthly cost.
  The scraper lives.

  TLS fingerprint, not IP reputation, was the dominant block all along.
