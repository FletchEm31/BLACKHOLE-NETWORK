  --    have this column before tonight, so existing rows will all default
  --    to 'N/A', which IS allowed. Listed here only for completeness.)
  SELECT DISTINCT edition FROM ebay_listings
   WHERE edition NOT IN ('1st Edition','Unlimited','Shadowless','N/A');

  -- 3. courtyard_sales / collector_crypt_sales sale_type — old CHECK
  --    allowed 'peer_to_peer','buyback','gacha'. Step 01 broadens to the
  --    v2 5-value set (fixed_price, auction, offer_accepted, buyback,
  --    peer_to_peer). 'gacha' is being DROPPED from the allowed set. If
  --    any row has sale_type='gacha', the new CHECK will reject. Check:
  SELECT 'courtyard_sales' AS tbl, sale_type, COUNT(*) FROM courtyard_sales
   WHERE sale_type='gacha' GROUP BY sale_type
  UNION ALL
  SELECT 'collector_crypt_sales', sale_type, COUNT(*) FROM collector_crypt_sales
   WHERE sale_type='gacha' GROUP BY sale_type;
  -- If 0 rows: safe. If any rows: decide UPDATE to peer_to_peer or new
  -- vocab value before applying step 01.

If all three queries return zero rows, the migration is shape-safe and
step 01 will apply cleanly.

NOTES ON DESIGN CHOICES
================================================================================

  1. Renames done via ALTER-then-RENAME (operator-confirmed). Preserves
     data, single transaction per table, PK/seq/index renames all carry
     through. Avoided the "build fresh + INSERT SELECT + DROP old" approach
     because it would risk dropped writes during the rename window.

  2. card_id FK was already wired by sql/card-id-resolver.sql earlier today
     across all 5 tables affected. Skipped re-adding it in step 01.

  3. courtyard_sales / collector_crypt_sales sale_type CHECK is the only
     constraint being NARROWED-then-WIDENED. Old CHECK ('peer_to_peer',
     'buyback','gacha') is dropped and replaced with the v2 5-value set.
     'gacha' is no longer in the allowed list — see the pre-apply check
     above for the verification query.

  4. fee_schedule unique key is (market, fee_name, effective_date). This
     lets a single fee_name carry multiple historical rates as platforms
     change — INSERT a new row with a later effective_date and let queries
     filter by it. Expired promos (e.g. the eBay FVF 50% promo with
     promo_expires_at='2026-05-04') are kept in the table for historical
     reference per spec.

  5. estimate_trade_costs() uses the spec's default $25 minimum-profit
     threshold for the is_profitable flag. Configurable in code if HORIZON
     needs a different threshold; not surfaced as a parameter today to keep
     the function signature stable.

  6. arbitrage_positions has signal_id REFERENCES tokenized_arbitrage_signals(id).
     This is the only FK between v2 tables — the rest stay soft-keyed by
     card_id only, per the existing data-standard §4 enforcement tiers.
