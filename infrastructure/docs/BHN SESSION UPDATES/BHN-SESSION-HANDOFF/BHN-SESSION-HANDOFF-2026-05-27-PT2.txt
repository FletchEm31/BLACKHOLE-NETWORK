NOTES ON DESIGN CHOICES
================================================================================

  1. Renames done via ALTER-then-RENAME (operator-confirmed). Preserves
     data, single transaction per table, PK/seq/index renames all carry
     through. Avoided the "build fresh + INSERT SELECT + DROP old" approach
     because it would risk dropped writes during the rename window.

  2. card_id FK was already wired by sql/card-id-resolver.sql earlier today
     across all 5 tables affected. Skipped re-adding it in step 01.

  3. courtyard_sales / collector_crypt_sales sale_type CHECK is the only
     constraint being NARROWED-then-WIDENED. Old CHECK ('peer_to_peer',
     'buyback','gacha') is dropped and replaced with the v2 5-value set.
     'gacha' is no longer in the allowed list — see the pre-apply check
     above for the verification query.

  4. fee_schedule unique key is (market, fee_name, effective_date). This
     lets a single fee_name carry multiple historical rates as platforms
     change — INSERT a new row with a later effective_date and let queries
     filter by it. Expired promos (e.g. the eBay FVF 50% promo with
     promo_expires_at='2026-05-04') are kept in the table for historical
     reference per spec.

  5. estimate_trade_costs() uses the spec's default $25 minimum-profit
     threshold for the is_profitable flag. Configurable in code if HORIZON
     needs a different threshold; not surfaced as a parameter today to keep
     the function signature stable.

  6. arbitrage_positions has signal_id REFERENCES tokenized_arbitrage_signals(id).
     This is the only FK between v2 tables — the rest stay soft-keyed by
     card_id only, per the existing data-standard §4 enforcement tiers.

ADDENDUM — APPLICATION TO LA + STEP 06 (FVF TIER FIX)

After the original handoff above was written, the operator authorized

──────────────────────────────────────────────────────────────────────────────
A. sold_listings.sale_type pre-existed with non-canonical values
──────────────────────────────────────────────────────────────────────────────

The pre-apply check (graders + edition + sale_type='gacha' on the tokenized
tables) all returned 0 rows — the migration looked shape-safe. Step 01
applied cleanly through 21 ALTER TABLEs, then errored on:

    ERROR: check constraint "sold_listings_sale_type_chk" of relation
           "sold_listings" is violated by some row

Diagnosis: sold_listings ALREADY had a sale_type column (not net-new as
the spec assumed; ADD COLUMN IF NOT EXISTS quietly skipped it). It carried
651 rows with eBay-native values: Auction (328), Best Offer (181), BIN (142).
The new v2 CHECK rejected them.

Resolution: one-shot UPDATE block to normalize:
    'Auction'    → 'auction'          (case-fold, 328 rows)
    'BIN'        → 'fixed_price'      (Buy It Now, 142 rows)
    'Best Offer' → 'offer_accepted'   (a SOLD row with sale_type='Best Offer'
                                       means seller accepted the offer, 181)

Mapping is semantically lossless. Re-applied step 01 successfully — all 5
renames + 13 CHECKs landed. Transaction wrapping meant the partial first
attempt left no half-applied state.

Carry-forward: sold_listings carried more pre-existing columns than the
spec accounted for (card_number, edition, sale_type already existed). The
IF NOT EXISTS guard handled the column adds, but the CHECK addition needed
the data normalization above. Future migrations against this table should
inspect live schema first (\d in psql) rather than trust the spec's
"these are net-new" framing.

──────────────────────────────────────────────────────────────────────────────
B. estimate_trade_costs() double-counted eBay FVF tiers (step 06)
──────────────────────────────────────────────────────────────────────────────

Steps 02–05 applied without issue. While verifying step 04 against Part 2
§4's worked example (courtyard_to_ebay, $800 buy / $1,100 sell, expected
~$170 sell_fees and +$121 net), the function returned $313.90 sell_fees
and -$23.91 net — NOT profitable. Same trade, 2x over the spec's expected
sell fees.

Diagnosis: fee_schedule seeded BOTH eBay FVF tiers as separate rows:
  'Final Value Fee'   (non-store, 13.25%)
  'FVF Basic Store'   (basic_store, 12.35%)
They're mutually exclusive — a seller is on ONE plan, not both — but
nothing in the schema marked them as such. The function summed both,
producing $145.75 + $135.85 = $281.60 just in FVF before payment processing.

Fix (sql/market-data-standard-06-fee-tier-fix.sql, applied + committed):

  1. ALTER fee_schedule ADD COLUMN tier TEXT (nullable) with CHECK
     {all, non_store, basic_store, premium_store, anchor_store}.

  2. Tag existing rows:
       non_store    Final Value Fee, FVF Above $7,500           (2)
       basic_store  FVF Basic Store, FVF Basic Above $2,500     (2)
       all          Payment Processing, Per-Order (both), AG,
                    Shipping (graded card), FVF 50% Promo       (6)
       NULL         every Courtyard / Collector Crypt row       (9)
                                                       ────────
                                                       Total    19

  3. DROP + CREATE estimate_trade_costs with new signature:
       estimate_trade_costs(p_buy_market, p_sell_market, p_buy_price,
                            p_sell_price, p_direction,
                            p_ebay_tier DEFAULT 'non_store')
     Filter clause on both buy-side and sell-side fee aggregation:
       AND (tier IS NULL OR tier = 'all' OR tier = p_ebay_tier)

     Signature changed (added a 6th parameter), so CREATE OR REPLACE
     wasn't sufficient — DROP + CREATE was required. EXECUTE grants
     reapplied against the new signature.

Verification on LA:
  non_store tier:   sell_fees $178.05, net +$111.94, ROI 13.99%, profitable ✓
  basic_store tier: sell_fees $168.15, net +$121.84, ROI 15.23%, profitable ✓

The spec's "$121 net / 15.2% ROI" target matched basic_store EXACTLY —
the spec authors implicitly used Basic Store rates. Default to non_store
gives the conservative number; trades profitable at non_store are
profitable at any higher tier.

              Uses the same App ID as the vintage scrapers

  - Parser: extracts item_id, sold_price, sold_at (endTime), sale_type

            both NULL if either can't be derived.
  - INSERT: WITH grade_check CTE consults master_grade_catalog FK before
            insert — non-canonical (grader, grade) pairs land as (NULL,

  1. Transfer(address from, address to, uint256 tokenId)
     - Standard ERC-721 transfer event
     - Captures every change of ownership (mint, sale, transfer, redemption)
  2. OrderFulfilled
     - Seaport (OpenSea) protocol-level event
     - Carries price, buyer, seller, royalty info
     - Filter to orders involving the Courtyard contract
