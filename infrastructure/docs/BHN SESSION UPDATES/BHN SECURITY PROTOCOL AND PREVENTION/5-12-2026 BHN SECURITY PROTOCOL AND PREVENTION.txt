PURPOSE

This document defines the official security testing, monitoring,

This is a living document. Update after every security event,

SECTION 1 — SECURITY PHILOSOPHY
==================================================================

BHN SECURITY PRINCIPLES:
  1. Defense in depth — multiple independent security layers
  2. Default deny — nothing allowed unless explicitly permitted
  3. Least privilege — every role gets minimum required access
  4. Zero trust — no node trusts another without verification
  5. Fail closed — if something breaks, it locks down not opens
  6. Dark hub — LA never exposes itself to internet directly
  7. Assume compromise — design as if any node can be breached

THREAT MODEL:
  Primary threats:
    - Automated port scanning and brute force (daily, constant)
    - Credential stuffing (if any public-facing services)
    - Supply chain attacks (compromised packages/images)
    - Provider-level compromise (Vultr/Hetzner insider)
    - Traffic analysis (correlation attacks on VPN)

  Secondary threats:
    - CVE exploitation (unpatched kernel/packages)
    - Docker escape (container → host)
    - WireGuard key compromise
    - DNS poisoning/hijacking
    - Social engineering of operator

  Out of scope (for now):
    - Nation-state adversaries
    - Hardware implants
    - Physical datacenter access

WHAT WE'RE PROTECTING:
  - Operator identity and location
  - PostgreSQL data (sessions, trades, memories)
  - WireGuard private keys
  - API credentials (Anthropic, Twilio, Alpaca)
  - Trading strategy logic and positions
  - HORIZON conversation history

SECTION 2 — SECURITY STACK (ALL NODES)
==================================================================

LAYER 1 — NETWORK PERIMETER (UFW):
  Default policy: deny incoming, deny outgoing, deny routed
  Only explicitly whitelisted ports/IPs allowed
  LA has the strictest ruleset (hub stays dark)
  After every new WireGuard peer: ALWAYS add egress UFW rules

LAYER 2 — INTRUSION PREVENTION (fail2ban):
  Bans IPs after repeated failed SSH attempts
  Whitelist: 127.0.0.1/8, ::1, 10.8.0.0/24 (WG tunnel), operator home IP
  Check status: fail2ban-client status
  Check bans: fail2ban-client banned

LAYER 3 — CROWD INTELLIGENCE (CrowdSec):
  AI-driven threat detection
  Shares threat intel with CrowdSec community
  Blocks known malicious IPs before they attempt anything
  Version: 1.7.8 (upgraded May 2026)
  Check: crowdsec-cli decisions list

LAYER 4 — DEEP PACKET INSPECTION (Suricata):
  IDS/IPS — inspects network traffic patterns
  Alerts on: port scans, exploit attempts, anomalous traffic
  Logs to: PostgreSQL node_logs table (via log_shipper)
  Check: systemctl status suricata

LAYER 5 — ENCRYPTED DNS (dnscrypt-proxy):
  All DNS queries encrypted (DoH/DNSCrypt)
  6 resolver rotation (Mullvad, Quad9, Cloudflare, AdGuard, NextDNS, Digitale)
  No plaintext DNS ever leaves any BHN node
  Check: dig +short cloudflare.com @127.0.0.1

LAYER 6 — SSH HARDENING:
  Key-only authentication (no passwords)
  PermitRootLogin=prohibit-password (LA)
  NJ uses non-standard port 2222
  Host keys regenerated during bootstrap

LAYER 7 — WIREGUARD + PSK:
  All inter-node traffic encrypted
  PSK adds quantum-resistant layer
  Silent on unauthenticated packets
  No response to probe packets

LAYER 8 — STORAGE ENCRYPTION (LA only):
  LUKS2 on NVMe and HDD
  Auto-unlock via keyfiles at boot
  Backup passphrases in Proton Pass

LAYER 9 — TRAFFIC OBFUSCATION (exit nodes):
  Shadowsocks: VPN traffic looks like HTTPS
  Tor relay: cover traffic blends in
  tinyproxy: API calls appear from Hillsboro

SECTION 3 — DAILY SMOKE TESTS
==================================================================

Run these checks at the START of every session to confirm
the network is healthy before doing any work.

─────────────────────────────────────────────────────────────
SMOKE TEST 1 — WireGuard Mesh Health
─────────────────────────────────────────────────────────────
Run on PC LA CONSOLE:

  wg show wg0

Expected output — ALL peers should show:
  ✅ latest handshake: < 3 minutes ago
  ✅ transfer: bytes sent AND received (not 0)
  ✅ endpoint: correct public IP

FAIL conditions:
  ❌ No handshake (peer unreachable)
  ❌ Only sent, nothing received (one-way tunnel)
  ❌ Handshake > 10 minutes ago (stale/broken)

─────────────────────────────────────────────────────────────
SMOKE TEST 2 — LA Hub Health
─────────────────────────────────────────────────────────────
Run on PC LA CONSOLE:

  systemctl --failed
  sudo docker ps | grep n8n
  sudo -u postgres psql -d eventhorizon -c "SELECT NOW();"
  df -h /mnt/eh-nvme-hot
  df -h /mnt/eh-hdd-cold
  free -h

Expected:
  ✅ No failed systemd units
  ✅ n8n container running
  ✅ PostgreSQL responds
  ✅ NVMe < 80% full
  ✅ HDD < 80% full
  ✅ RAM not critically low

==================================================================

PURPOSE: Verify that security controls actually work.
Run quarterly or after major architecture changes.

⚠️ IMPORTANT: Only test YOUR OWN infrastructure.
   Never run these against external systems.
   Document all tests with timestamps.

─────────────────────────────────────────────────────────────
PENTEST 1 — External Port Scan
─────────────────────────────────────────────────────────────
From a NON-BHN IP (e.g. your home internet, not through VPN):

  nmap -sV -p 1-65535 149.28.91.100   (LA)
  nmap -sV -p 1-65535 192.248.187.208 (Frankfurt)
  nmap -sV -p 1-65535 5.78.94.237     (Hillsboro)
  nmap -sV -p 1-65535 140.82.4.35     (NJ)

Expected results:
  LA:         22/tcp open ssh (or filtered)
              51820/udp WireGuard (may show as filtered)

SECTION 7 — INCIDENT RESPONSE PROTOCOL
==================================================================

SEVERITY LEVELS:
  P1 CRITICAL  — Active breach, data exfiltration, key compromise
  P2 HIGH      — Unauthorized access attempt succeeding, service down
  P3 MEDIUM    — Anomalous traffic, failed breach attempt, CVE
  P4 LOW       — Informational, routine security events

─────────────────────────────────────────────────────────────
P1 CRITICAL — Active Breach Response
─────────────────────────────────────────────────────────────
IMMEDIATE (< 5 minutes):
  1. Isolate compromised node:
     ufw default deny incoming
     ufw default deny outgoing
     ufw reload

  2. Kill all WireGuard peers on compromised node:
     wg-quick down wg0

  3. On LA — remove compromised peer:
     wg set wg0 peer <PUBKEY> remove
     wg-quick save wg0

  4. Rotate WireGuard keys on LA:
     wg genkey > /etc/wireguard/private.key
     wg pubkey < /etc/wireguard/private.key > /etc/wireguard/public.key

  5. If trading node (NJ) — trigger killswitch:
     python3 /opt/bhn/scripts/trading/master_killswitch.py halt \
       --reason "Security incident - P1"

SHORT TERM (< 1 hour):
  6. Rotate all API keys (Anthropic, Twilio, Alpaca)
  7. Review PostgreSQL logs for data access
  8. Snapshot all remaining nodes for forensics
  9. Rebuild compromised node from scratch via bootstrap
  10. Review how breach occurred — patch the vector

─────────────────────────────────────────────────────────────
P2 HIGH — Unauthorized Access Attempt
─────────────────────────────────────────────────────────────
  1. Identify source IP:
     fail2ban-client banned
     grep "Failed\|Invalid" /var/log/auth.log | tail -50

  2. Manually ban if not already:
     fail2ban-client set sshd banip <SOURCE_IP>

  3. Check if CrowdSec caught it:
     crowdsec-cli alerts list | head -20

  4. Log the incident:
     sudo -u postgres psql -d eventhorizon -c "
     INSERT INTO security_events (event_type, severity, source_ip, description)
     VALUES ('unauthorized_access_attempt', 'high', '<IP>', 'Manual P2 incident log');"

  5. Monitor for 24h — escalate if continues

─────────────────────────────────────────────────────────────
P3 MEDIUM — Anomalous Traffic / Failed Breach
─────────────────────────────────────────────────────────────
  1. Check Suricata logs:
     sudo cat /var/log/suricata/fast.log | grep <SOURCE_IP>

  2. Check CrowdSec:
     crowdsec-cli decisions list | grep <SOURCE_IP>

  3. Review UFW logs:
     grep UFW /var/log/syslog | tail -50

  4. Log and monitor — no immediate action if blocked

─────────────────────────────────────────────────────────────
P4 LOW — Routine Security Event
─────────────────────────────────────────────────────────────
  - Let automated systems handle (fail2ban, CrowdSec)
  - Review in weekly security check
  - No immediate action required

ROTATION PROCEDURE (WireGuard keys — example):
  1. Generate new key on node:
     wg genkey > /etc/wireguard/private.key.new

  2. Update LA wg0.conf with new pubkey for that peer

  3. Replace key on node:
     mv /etc/wireguard/private.key.new /etc/wireguard/private.key

  4. Restart WireGuard on both sides within 60 seconds

  5. Update Proton Pass with new key values
