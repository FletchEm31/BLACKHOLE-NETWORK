Bug 2: bhn-vnstat-collector.sh malformed timestamps          (c679bab)
------------------------------------------------------------------
Two root causes:
  (a) jq operator precedence — `array | map() + array | map() + ...`
      bound | tighter than +, so three of four period-type arrays were
      pipe-transformed away instead of concatenated.
  (b) Null date handling on "top" entries where vnstat hadn't yet
      recorded a top-day. jq's `@text` on null produced the literal
      string "null", yielding the timestamp `null-null-1T00:00:00Z`,
      rejected by PG.

Fix: parenthesised each `(array | map(...))` block so + does array
concat first; added `select(.d != null and .d.year != null and
.d.month != null)` to drop incomplete entries.

Secondary issue uncovered while testing the fix — `log_shipper` had
INSERT but not UPDATE on node_bandwidth_stats. INSERT ... ON CONFLICT
DO UPDATE requires both privileges. PG returned a misleading
"permission denied for table" error. Granted UPDATE on 12 mutable
infrastructure collector tables to log_shipper on LA:
  node_bandwidth_stats, iptables_stats, container_stats,
  pg_activity_snapshots, n8n_execution_stats, fail2ban_events,
  dns_query_log, wg_peer_stats, connection_snapshots,
  node_resource_stats, tor_relay_stats, proxy_request_logs.

Verified: LA wrote 577 rows, NJ wrote 256 rows across hour/day/month/top
buckets. Timestamps clean (2026-05-13 through 2026-05-27). Frankfurt
still needs `apt install vnstat` (separate apt-level issue, not script).
collector_config row 'Node Bandwidth (vnstat)' flipped to ACTIVE.

Bug 3: bhn-security-events-collector.sh invalid inet         (31f0303)
------------------------------------------------------------------
Root cause: the three IP-extraction regexes (UFW SRC=, SSH "from", and
fail2ban "Ban") used `[0-9.]+`, which matched malformed log entries
like `SRC=0000` in UFW logs around 2026-05-24 19:25 UTC. PG rejected
'0000'::inet, killing the batch INSERT, leaving the state file frozen
at 2026-05-24T19:25:07. Cron kept firing for three days, silently
failing each time.

Fixes:
  - Tightened all three regexes to require a real IPv4 shape:
      ([0-9]{1,3}\.){3}[0-9]{1,3}
  - Added a defensive valid_ipv4() helper invoked on each captured IP.
    If something still slips through, the row inserts with source_ip
    = NULL rather than killing the batch. Helper explicitly `return 0`
    so a no-match doesn't fire `set -e`.

IPv6 source IPs are not captured (regexes are IPv4-only). Same
behaviour as before; adding IPv6 capture is a separate follow-up.

Re-enabled /etc/cron.d/bhn-security-events (was renamed to .disabled-
2026-05-27 during Phase 1 diagnosis). Single invocation drained the
three-day backlog:
  11,071 rows inserted (5,268 ufw_block / 60 ssh_failure / 14 fail2ban)
  MAX(detected_at) caught up to 2026-05-27 04:45:06.
collector_config row 'Suricata/CrowdSec' flipped from OUTAGE to ACTIVE.

Memory updated: project_collector_script_bugs_2026_05_27.

Bug 4: bhn-conntrack-collector.sh — diagnosed, NOT fixed
------------------------------------------------------------------
The earlier brief framed this as "rows landing since May 24 with zero
metric columns". Reality: connection_snapshots table is **empty**.
No rows from any node.

Root cause: kernel built without `CONFIG_NF_CONNTRACK_PROCFS`. The
nf_conntrack module is loaded and tracking (61 active connections via
/proc/sys/net/netfilter/nf_conntrack_count), but the per-connection
/proc/net/nf_conntrack interface the script reads doesn't exist on
this kernel. Script's `[[ -r "$CONNTRACK_FILE" ]] || exit 0` guard
correctly handles the absence — exits clean with no rows, deploy
verify_run sees 0 delta, marks NO-DATA.

Fix paths (any one — bigger scope than vnstat/security-events):
  1. apt install conntrack-tools on each node; rewrite collector to
     use `conntrack -L`. Closest to current design.
  2. Rewrite to use `ss -tnH` (no install needed, more portable).
  3. Drop per-connection detail, report aggregate stats only.

Updated collector_config note with the kernel diagnosis. Decision
deferred to next session.
