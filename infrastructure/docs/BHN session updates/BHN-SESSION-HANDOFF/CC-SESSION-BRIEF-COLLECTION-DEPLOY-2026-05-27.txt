CONTEXT — WHAT'S BROKEN AND WHY
================================================================================

We did a full audit of BHN's data collection pipeline today. The Grafana Data
Health dashboard is live and showing the real picture: most infrastructure
collectors are either never deployed or stopped. The tables exist in PostgreSQL
(eventhorizon on LA, 10.8.0.1), the collector scripts exist in the repo, but
the cron jobs were never wired up on the nodes.

Additionally, the Grafana collector_config table (which drives the status
column on the Data Health dashboard) needs a new PENDING status so we can
distinguish between:
  - ACTIVE   = collecting, within cadence (green)
  - OUTAGE   = was collecting, now stopped (red)
  - MAINTENANCE = intentionally paused (yellow)
  - PENDING  = schema exists, collector never deployed (blue/grey)

Right now PENDING sources show as OUTAGE which is misleading.

TASK 1 — collector_config: add PENDING status
================================================================================

File: sql/collector-config-schema.sql

1. ALTER the CHECK constraint on status_override to allow 'pending':
   Currently: CHECK (status_override IN (NULL, 'maintenance'))
   Change to:  CHECK (status_override IN ('maintenance', 'pending'))
   Note: NULL check constraints don't work like that in PG — the correct
   syntax is: CHECK (status_override IS NULL OR status_override IN ('maintenance', 'pending'))

2. Seed the known PENDING sources by adding UPDATE statements at the bottom
   of the SQL file (after the INSERT block) to set status_override = 'pending'
   for collectors that have never been deployed:

   Sources to mark PENDING (schema exists, no cron, never collected):
     - 'n8n Executions'        (n8n_execution_stats — SQLite→PG bridge not built)
     - 'CrowdSec'              (crowdsec_decisions — collector not wired)
     - 'WireGuard Stats'       (wg_peer_stats — cron not deployed)
     - 'Node Resources'        (node_resource_stats — cron not deployed)
     - 'Docker Stats'          (container_stats — cron not deployed)
     - 'fail2ban'              (fail2ban_events — cron not deployed)
     - 'HORIZON Tokens'        (agent_token_log — HORIZON workflow broken)

   These were COLLECTING but stopped — leave as NULL (auto-computes to OUTAGE):
     - 'Suricata/CrowdSec'     (security_events — stopped May 24 19:25 UTC)
     - 'EIA Energy Prices'     (energy_prices — stale 12+ days)
     - 'USDA Agriculture'      (agriculture_prices — dead since January)

3. The UPDATE block should use ON CONFLICT logic — i.e. only set pending if
   status_override IS NULL (don't overwrite an existing maintenance flag):
   UPDATE collector_config
   SET status_override = 'pending'
   WHERE source_name IN (...)
   AND status_override IS NULL;

Apply on LA after editing:

   ssh root@10.8.0.1 "sudo -u postgres psql -d eventhorizon -f /tmp/collector-config-schema.sql"

TASK 5 — db_size_log table (quick win, 2 minutes)
================================================================================

Create the table and cron for DB size history tracking:

  sudo -u postgres psql -d eventhorizon -c "
  CREATE TABLE IF NOT EXISTS db_size_log (
    recorded_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    size_bytes   BIGINT NOT NULL
  );
  GRANT SELECT ON db_size_log TO grafana_reader, agent_reader, ehuser;"

  echo "0 0 * * * root sudo -u postgres psql -d eventhorizon -c \"INSERT INTO db_size_log (size_bytes) VALUES (pg_database_size('eventhorizon'));\"" \
    > /etc/cron.d/bhn-db-size-log

This enables the DB size over time chart on the Data Health dashboard.
