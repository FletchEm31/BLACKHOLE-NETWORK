Confirmed missing/incomplete:
1. Visual Crossing historical actuals — in progress (backfill cron running)

This is the biggest gap. Without 2-3 years of historical tmax for Denver and Miami, the calibration has no baseline to compare against. The cron is running nightly pulling 500 days at a time.
2. ECMWF 51-member ensemble — scaffolded, not collecting

This would give you the best probabilistic uncertainty data — 51 model runs showing the spread of possible outcomes. Currently Open-Meteo is approximating this but it's not the same. Requires pip install ecmwf-opendata cfgrib eccodes and GRIB2 parsing logic.
3. HRRR (High-Resolution Rapid Refresh) — scaffolded, not collecting

The 3km resolution short-range model. Most valuable in the 0-48 hour window right before contract settlement. Requires pip install herbie-data.
4. NWS gridpoints two-step API — scaffolded, not collecting

The actual NWS gridpoint forecast is scaffolded but the two-step discovery (/points → /gridpoints) isn't implemented. This is the most important one since Kalshi settles on NWS CLI reports — using the same NWS office's forecast is the most direct signal.
5. Outcome data — incomplete

The ledger can only compute outcomes for past contracts where weather_bronze_nws_actuals has the settled temperature. You only have 1,075 actuals rows — that's roughly 134 days of data for 8 cities. Denver and Miami go back to June 11 so you have ~2 weeks of outcomes to backfill.
