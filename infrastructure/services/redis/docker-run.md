# BHN HORIZON Redis — working-memory cache

Hot working-memory tier for HORIZON conversations. Holds the last N turns + running summary of every live session, keyed by `horizon:sess:<session_id>:*`. **Cache only** — durable state lives in PostgreSQL (`memories` + `conversation_sessions`). Persistence (AOF/RDB) is intentionally off.

**LA-only.** Bound to LA's WG tunnel (`<BHN_WG_LA_IP>:6379`). Public eth0 never exposes 6379. The HORIZON n8n workflow connects from the same host over the loopback-equivalent WG bind.

Pairs with `sql/redis-memory-schema.sql` (table `conversation_sessions`) and Phase 5.5 of `infrastructure/docs/horizon-roadmap.md`.

## Prerequisites

- Docker + docker compose on LA
- WG tunnel up (LA = `<BHN_WG_LA_IP>` on wg0)
- Proton Pass entry **`EH-Redis-Password`** populated with a strong random (generated in-conversation at deploy time)
- `sql/redis-memory-schema.sql` already applied to the `eventhorizon` database

## Deploy

```bash
# 1. Create the service dir on LA
sudo mkdir -p /opt/bhn-horizon-redis
cd /opt/bhn-horizon-redis

# 2. Copy docker-compose.yml + .env.example from repo
#    (scp from operator-pc, or git pull on LA if the repo is mirrored)
cp .env.example .env

# 3. Paste the password from Proton Pass into .env
#    Edit .env, replace REPLACE_ME with the EH-Redis-Password value
nano .env
chmod 600 .env

# 4. Bring it up
docker compose up -d
docker compose logs --tail 30

# 5. PING test (uses password from env)
set -a; source .env; set +a
docker exec bhn-horizon-redis redis-cli -a "$REDIS_PASSWORD" PING
# Expect: PONG
```

## VPN-only verification

```bash
# On LA, after deploy

# Expect: LISTEN on <BHN_WG_LA_IP>:6379  (NOT 0.0.0.0:6379)

# Public IP must NOT serve:
curl --max-time 3 telnet://<BHN_LA_PUBLIC_IP>:6379 2>&1 | head -3
# Expect: connection refused / timeout

# From the n8n container on LA (over the docker-host network):
docker exec -i bhn-n8n sh -c 'apk add --no-cache redis >/dev/null 2>&1; redis-cli -h <BHN_WG_LA_IP> -a "$REDIS_PASSWORD" PING'
# Expect: PONG
```

## UFW

No UFW change required. `<BHN_WG_LA_IP>:6379` is on the WG interface (wg0), not on eth0. UFW's existing wg0 ACCEPT rule covers it. Confirm with:

```bash
sudo ufw status verbose | grep -E '(wg0|6379)'
```

If the host has any explicit DENY/REJECT rule for 6379 on wg0, drop it. Otherwise no edits needed.

## n8n credential

Add a new credential in the HORIZON n8n instance:

| Field | Value |
|-------|-------|
| Name | `EH HORIZON Redis (LA)` |
| Type | Redis |
| Host | `<BHN_WG_LA_IP>` |
| Port | `6379` |
| Password | (paste from Proton Pass — same `EH-Redis-Password` value) |
| Database | `0` |
| SSL | off |

Test the credential in n8n — it should return "Connection successful".

## Redis key shape (reference)

For each live session HORIZON maintains three keys (all TTL = 12h sliding):

| Key | Type | Contents |
|-----|------|----------|
| `horizon:sess:<session_id>:turns` | list | LPUSH'd turn JSON, LTRIM'd to last 20 |
| `horizon:sess:<session_id>:summary` | string | Re-summarized every 5 turns via Haiku |
| `horizon:sess:<session_id>:meta` | hash | `channel`, `started_at`, `last_seen` |

`<session_id>` is the UUID generated by n8n at session start; same UUID is the PK in `conversation_sessions`.

## Resource footprint

- Image: ~30 MB (redis:7-alpine)
- RAM: ~30 MB idle, bounded by `--maxmemory 128mb` (LRU eviction if hit — safe because PG is source of truth)
- Disk: 0 bytes persistent (no AOF, no RDB)

Comfortably fits LA's remaining ~2 GB headroom.

## Backup

**None.** Redis here is intentionally ephemeral. If the container loses state, the worst case is that any live session loses its in-progress working memory — the next operator turn opens a new session and pgvector still recalls long-term context from `memories`. No data loss; just a degraded turn until the new session warms up.

## Operational notes

- **Password rotation**: generate a new value in Proton Pass, paste into `/opt/bhn-horizon-redis/.env`, `docker compose up -d` (recreates container with new password), update the n8n credential. Live sessions in Redis at rotation time are lost — acceptable, see "Backup".
- **Monitoring**: covered by the existing `bhn-docker-stats-collector` cron (container CPU/RAM lands in `container_stats`). No new monitoring needed.
- **redis_hit_ratio**: HORIZON's morning briefing (M2) eventually surfaces working-memory hit ratio as a health signal — captured by the session-close sweep workflow, not by Redis-side metrics.

## Troubleshooting

```bash
# Container won't start
docker compose logs redis | tail -50

# Auth failures from n8n
docker exec bhn-horizon-redis redis-cli -a "$REDIS_PASSWORD" CONFIG GET requirepass
# Expect: 1) "requirepass"  2) "<the password>"

# See live keys
docker exec bhn-horizon-redis redis-cli -a "$REDIS_PASSWORD" KEYS 'horizon:sess:*'

# Inspect a session
docker exec bhn-horizon-redis redis-cli -a "$REDIS_PASSWORD" HGETALL horizon:sess:<uuid>:meta
docker exec bhn-horizon-redis redis-cli -a "$REDIS_PASSWORD" LRANGE horizon:sess:<uuid>:turns 0 -1
docker exec bhn-horizon-redis redis-cli -a "$REDIS_PASSWORD" GET    horizon:sess:<uuid>:summary

# Memory pressure check
docker exec bhn-horizon-redis redis-cli -a "$REDIS_PASSWORD" INFO memory | grep -E '(used_memory_human|maxmemory_human|evicted_keys)'
```
