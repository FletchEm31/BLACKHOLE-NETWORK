─── SSH ACCESS ───────────────────────────────────────────────────────

From operator PC (direct to public IPs):

  PC LA CONSOLE      ssh root@149.28.91.100
  PC GE CONSOLE      ssh root@192.248.187.208
  PC NJ CONSOLE      ssh -p 2222 root@140.82.4.35

From operator PC via VPN tunnel (any wg profile):

  ssh root@10.8.0.1                              # LA hub via tunnel

From LA hub (orchestrator origin for cross-region admin):

  ssh frankfurt                                  # → root@10.9.0.2:2222 (alias in /root/.ssh/config)
  ssh nj                                         # → root@10.8.0.5:2222 (alias in /root/.ssh/config)

REMOTE BROWSER WINDOW (Vultr noVNC) is the emergency-only fallback when
SSH/tunnel is down — accessed from the Vultr web panel per VPS.

─── WIREGUARD ROUTING ────────────────────────────────────────────────

Hub: LA wg0 listens on UDP 51820.

Peers:
  10.8.0.2      FLETCH-PHONE                       split + full profiles
  10.8.0.4      FLETCH-DESKTOP                     split (EH-admin) + full (EH-full) profiles
  10.8.0.5      NJ trading node                    underlay 140.82.4.35:51820
  10.9.0.2      Frankfurt exit                     underlay 192.248.187.208:51821, keepalive 25s

Frankfurt exit routing (APPLIED 2026-05-12 but BROKEN — deferred to next session):
  LA-side policy routing is wired correctly (fwmark 0x100 → table 100,
  default via 10.9.0.2 dev wg0 onlink) but Frankfurt is missing the
  MASQUERADE NAT rule for the 10.8.0.0/24 source. Hub-client packets
  reach FRA over the tunnel; FRA can't rewrite the source IP for return
  traffic; full-tunnel internet appears dead.

  Next-session work:
    On FRA:  iptables -t nat -A POSTROUTING -s 10.8.0.0/24 -o enp1s0 -j MASQUERADE
             iptables-save > /etc/iptables/rules.v4
    On LA:   /etc/wireguard/bhn-frankfurt-exit.sh apply
    Verify:  curl https://api.ipify.org   (from full-tunnel profile)
             → expected: 192.248.187.208

  Current state: rolled back on LA via `bhn-frankfurt-exit.sh rollback`.
  Script: /etc/wireguard/bhn-frankfurt-exit.sh (apply | rollback | status).


─── DATABASES ────────────────────────────────────────────────────────

PostgreSQL on LA (NVMe, LUKS2-encrypted):
  Database name: `eventhorizon` (intentional preservation — see below)
  VPN-only bind at 10.8.0.1:5432
  Roles + their access:
    postgres           superuser, peer auth via local socket
    ehuser             rw for legacy collectors
    n8n_user           rw for n8n workflows
    agent_reader       read-only for HORIZON / AI Agent
    grafana_reader     read-only for Grafana dashboards
    bhn_trader         rw for trading framework (per scripts/trading/)
    log_shipper        INSERT-only on node_logs (used by FRA + future nodes)


─── DNS ──────────────────────────────────────────────────────────────

dnscrypt-proxy on LA bound to 127.0.0.1 + 10.8.0.1 (port 53).
Upstreams: Cloudflare, Quad9, Mullvad, NextDNS (DoH/DNSCrypt over 443).
VPN clients on "full" profile use DNS=10.8.0.1 to prevent DNS leak.

product. THIS infrastructure is BHN, personal-only.


─── EMERGENCY PROCEDURES ─────────────────────────────────────────────

Trading framework killswitch (halt all 5 strategies):
  ssh nj 'python3 /opt/bhn/trading/master_killswitch.py halt --reason "..."'

Reset trading framework (re-arm after halt — operator decision):
  ssh nj 'python3 /opt/bhn/trading/master_killswitch.py reset --confirm'

Frankfurt exit routing rollback:
  ssh la /etc/wireguard/bhn-frankfurt-exit.sh rollback

WireGuard hub down (LA):
  REMOTE BROWSER WINDOW → systemctl restart wg-quick@wg0

n8n DB corruption (verified 2026-05-09 incident, runbook in memory):
  See reference_n8n_db_recovery memory — sqlite3 .recover → temp DB → hot-swap
